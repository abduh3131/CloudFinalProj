ABDULLAH
Role: Full Code Walkthrough (the whole codebase)
~5-6 minutes
============================================================

[WEBCAM ON, face visible]

"Hey so my name is Abdullah, and this is our Phase 1 demo for
cloud computing. So what we built is basically a Python
application that takes real traffic data from the NGSIM dataset,
this is vehicle trajectory data that was recorded on the US-101
freeway in Los Angeles, and our system goes through all of it
and detects three types of driving scenarios: car-following,
stop-and-go, and lane changes. My part was building the core
codebase so I'm gonna walk through the entire thing and explain
how it all works."

[OPEN VS Code with project folder expanded in sidebar]

"So here's the project structure. We went with a modular
monolithic architecture, so it's one application but the code
is organized into separate modules. At the top you can see
main.py which is the entry point, config.py which has all our
settings and thresholds, and then the modules folder is where
all the actual logic is."


--- CONFIG.PY ---

[OPEN config.py, scroll slowly]

"Let me start with config. This is where everything is
configured in one place."

[SCROLL TO: the "AZURE BLOB STORAGE SETTINGS" comment block]

"At the top here we have the Azure settings. The connection
string comes from a .env file for security, and we have two
containers, ngsim-raw for the raw data and ngsim-output for
the results."

[SCROLL TO: the "DATASET SETTINGS" comment block]

"Then the dataset settings. The NGSIM data is sampled at 10
hertz, so 10 frames per second. Our window size is 5 seconds,
so that gives us 50 frames per window. That's the size of each
chunk we analyze."

[SCROLL TO: the "CAR-FOLLOWING DETECTION THRESHOLDS" comment block]

"And then here are all the detection thresholds. For
car-following, the max headway is 200 feet, the minimum speed
to count as moving is 5 feet per second, and the max speed
difference between the two cars is 15 feet per second."

[SCROLL TO: the "STOP-AND-GO DETECTION THRESHOLDS" comment block]

"For stop-and-go, the decel threshold is negative 5, the accel
is 3, and the low speed threshold is 10 feet per second which
is about 7 miles an hour, basically crawling."

[SCROLL TO: the "LANE CHANGE DETECTION THRESHOLDS" comment block]

"And lane change just checks mainline lanes 1 through 5 and
uses a 200 foot radius to find surrounding vehicles. All of
this is configurable in this one file."


--- MAIN.PY ---

[OPEN main.py]

[SCROLL TO: the "MAIN PIPELINE" comment block in the main() function]

"Main.py runs a 5-step pipeline, you can see it right here in
the comments. Step 1 is data ingestion, loading the CSV or
downloading from Azure. Step 2 is preprocessing, cleaning the
data. Step 3 is scenario detection where it runs all three
detectors. Step 4 formats and saves the output. And Step 5
generates the visualization charts."

[SCROLL through each STEP comment block briefly]

"And you can see each step is clearly marked. It supports
command line flags too, like --data to point to a specific
file and --azure to enable the cloud upload."


--- INGESTION.PY ---

[OPEN modules/ingestion.py]

[SCROLL TO: "LOAD NGSIM DATA FROM LOCAL CSV FILE" comment block]

"Ingestion handles loading the data. The main function here is
load_ngsim_local. It reads the CSV, and then it has to
normalize the column names because the data from the government
website uses slightly different names than the standard format."

[SCROLL TO: "NORMALIZE COLUMN NAMES" comment block]

"You can see here it maps things like Space_Headway to
Space_Hdwy, v_vel to v_Vel. It also drops extra columns we
don't need and filters to only US-101 data since the dataset
can have multiple freeways."

[SCROLL TO: "UPLOAD RAW DATA TO AZURE" comment block]

"And then here's the Azure integration. Upload raw data takes
the CSV and pushes it to our ngsim-raw blob container. Download
pulls it back. This uses the azure-storage-blob Python SDK."


--- PREPROCESSING.PY ---

[OPEN modules/preprocessing.py]

[SCROLL TO: "MAIN PREPROCESSING PIPELINE" comment block]

"Preprocessing cleans everything up. You can see the steps
listed right here in the comments. It filters to mainline lanes
1 through 5 only, because ramps and auxiliary lanes would give
false detections. It removes sensor errors, like if the
acceleration is over 40 feet per second squared that's obviously
bad data. And then it computes some extra features."

[SCROLL TO: "COMPUTE DERIVED FEATURES" comment block]

"Down here it adds speed in miles per hour, a relative
timestamp, and counts how many frames each vehicle appears in.
After all this we go from like 400,000 raw records down to
about 189,000 clean ones."


--- CAR FOLLOWING DETECTOR ---

[OPEN modules/scenarios/car_following.py]

[SCROLL TO: "MAIN DETECTION FUNCTION - CAR FOLLOWING" comment block]

"Now the scenario detectors, this is the core of the whole
thing. All three use the same sliding window approach. You take
each vehicle's data, sort it by time, and slide a window of 50
records across it."

"For car-following, you can see the checks are numbered in the
comments. Check 1, the ego vehicle stays in the same lane for
the whole window."

[SCROLL slowly through the CHECKs]

"Check 2, it has the same car in front of it, a consistent
leader. Check 3, the lead vehicle is also in the same lane.
Check 4, the space headway is within 0 to 200 feet. Check 5,
both vehicles are actually moving, not parked. And Check 6,
the speed difference is less than 15 feet per second so they're
going about the same speed. If all six pass, that's a
car-following scenario."


--- STOP AND GO DETECTOR ---

[OPEN modules/scenarios/stop_and_go.py]

[SCROLL TO: "MAIN DETECTION FUNCTION - STOP AND GO" comment block]

"Stop-and-go looks for three phases in order. You can see
Check 1 here, the window has to contain all three: hard braking,
a near-stop low speed, and re-acceleration."

[SCROLL TO CHECK 2]

"Check 2 makes sure the deceleration happens before the
acceleration. So it has to be in the right order: brake first,
slow down, then speed up."

[SCROLL TO: "CONGESTION VERIFICATION" comment block]

"And then this is an important one. The congestion verification
checks that nearby vehicles are also going slow or braking. That
way we know it's actual traffic congestion and not just one
random car hitting their brakes."


--- LANE CHANGE DETECTOR ---

[OPEN modules/scenarios/lane_change.py]

[SCROLL TO: "MAIN DETECTION FUNCTION - LANE CHANGE" comment block]

"Lane change is more straightforward. Check 1, does the Lane_ID
change during the window. Check 2, is it an adjacent lane, so
the difference is exactly 1. Check 3, both lanes have to be
mainline lanes, 1 through 5, no ramps."

"It also records the source lane, destination lane, the
direction of the change, and finds surrounding vehicles in both
lanes."


--- WRAP UP ---

"So yeah that's the full codebase. Everything is modular, each
file has one job, and the comment blocks make it easy to
navigate. We ended up detecting 1,923 total scenarios from the
real NGSIM data. Now my teammates will show the system running,
the Azure storage, and the visualizations."
